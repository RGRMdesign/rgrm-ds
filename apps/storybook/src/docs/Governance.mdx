# Governance

How the design system is structured, versioned, and extended.

## Architecture principles

- **One package per layer, not per component.** Adding a component never adds a package;
  it lives inside the existing `css`, `core`, `react`, and `elements` packages.
- **CSS is the single source of truth for visuals.** React and Web Component layers are
  thin wrappers that apply the same BEM classes — never re-style on top.
- **Tokens are a peer dependency.** Packages only _use_ token custom properties
  (`var(--font-size-h1)`, …); the token set is loaded once by the consuming app.
- **Tree-shakeable sub-path exports.** Each component is reachable via its own entry
  (e.g. `@rgrmdesign/rgrm-ds-react/heading`) plus a barrel that re-exports everything.

## Versioning & releases

- Packages follow **semantic versioning** and are published under the `@rgrmdesign`
  scope on npm.
- The CSS, React, and Web Component packages are released together so a component is
  available in every layer at the same version.
- **Core is published too.** `@rgrmdesign/rgrm-ds-core` is a runtime dependency of the
  React and Web Component packages (it is not bundled into them), so it must be available
  on npm as a transitive dependency. Most apps never install it directly; release it in
  lockstep with the wrappers that depend on it.
- Design tokens live in a **separate repository** (generated from <a href={FIGMA_LIBRARY_URL}>Figma variables</a>) and are consumed here as a peer dependency (`>=0.13.1`).
- Storybook is the published reference and deploys from `main` to GitHub Pages:
  **https://rgrmdesign.github.io/rgrm-ds/**

## Definition of done for a component

A component is considered complete when it exists across all relevant layers with
matching docs:

1. **Core** — `classNames()` at minimum (see Link); variant types or shared constants when needed (see Button).
2. **CSS** — `packages/css/src/<name>/index.css`, imported from the package barrel, with
   a build step and a `./<name>` export.
3. **React / Elements** — wrappers that re-use the core logic and CSS classes, each with
   a `./<name>` sub-path export.
4. **Sandboxes** — a demo in all three sandbox apps (`sandbox-css`, `sandbox-react`,
   `sandbox-element`).
5. **Storybook** — Overview plus CSS, React, and Web Components docs and stories,
   `fixtures.ts`, `snippets.ts`, and a `storySort` entry in `preview.tsx`.
6. **Release** — changeset (minor for new components) and roadmap table updated when the
   component ships.

## Accessibility

- Use semantic HTML; wrappers expose the right element (e.g. heading levels map to
  `<h1>`–`<h6>`).
- Colour and contrast come from the token themes — verify new combinations meet WCAG AA.
- Every theme (Root / Dark / Brand) must remain usable; preview all three via the
  Storybook toolbar.

## Contributing

- Visual changes start in the <a href={FIGMA_LIBRARY_URL}>Figma library</a>; token updates
  flow through the token repository before landing in code.
- Use **Node.js 24.x** (see `.nvmrc`) and the pinned **pnpm** version (`packageManager`
  in the root `package.json`).
- Run `pnpm format` (Prettier) before committing.
- Add or update Storybook docs and all three sandbox demos alongside any component change.
- See `README.md` (Adding a new component) and `.cursor/commands/build-component.md` for the
  full checklist.
